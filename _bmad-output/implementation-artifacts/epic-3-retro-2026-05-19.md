## Défis

1. **Sécurité documents : deux CR blockers corrigés in-story** — Story 3.2 a eu deux findings sécurité critiques (P1 : métadonnées documents exposées aux non-auteurs, P2 : confiance excessive dans les clés R2 client). Les deux ont été patchés pendant la review. Ces patterns sont des pièges récurrents dans les apps avec upload de fichiers.

2. **`&&` JSX en ternaire conditionnelle (Story 3.4)** — Pas le pattern classique `&&` dans JSX, mais `!isAuthor && !isAdmin ? ...` — le `&&` dans la condition d'une ternaire. Le guardrail Next.js 16 strict attrape aussi ce pattern. Solution : précalculer le booléen en `const shouldShowWhatsApp = !isAuthor && !isAdmin`.

3. **`currentAdminApproved` perdu après merge optimiste (Story 3.5)** — Le CR de la Story 3.5 a trouvé que l'update optimiste dans le Kanban admin perdait le champ `currentAdminApproved` après la première approbation double-vérification. Le `router.refresh()` ne réinitialise pas les `useState` initialisés depuis les props. Patch appliqué.

4. **JSON invalide sur `PATCH /verify` et action POST par défaut (Story 3.3)** — Le CR de la Story 3.3 a trouvé deux bugs : (a) JSON malformé retournait une erreur 500 au lieu de 400, (b) le POST de compatibilité interprétait toute action inconnue comme un rejet. Les deux patchés.

5. **`dev.db` binaire commité (Story 3.3)** — Un DS subagent a inclus le fichier base de données binaire dans son commit. Patché en revertant le fichier. Le `.gitignore` couvre `*.db` mais le git add `-A` du DS l'a quand même inclus.

6. **deferred-work.md accumule des items anciens** — Les items `avatarUrl vs image` et `Subscription.providerRef NOT NULL` de la Story 2.0 sont toujours ouverts. Ils ne bloquent pas mais s'accumulent.

## Insights Clés

1. **Les patterns sécurisés R2 sont reproductibles** — Le pattern presign → upload → complete avec validation côté serveur et restriction métadonnées est maintenant éprouvé. Toute story qui implique des uploads de fichiers devrait suivre ce modèle.

2. **Les reviews sécurité sont des blockers, pas des suggestions** — Les deux findings de sécurité de la Story 3.2 (exposition de métadonnées, confiance excessive dans les clés client) l'ont prouvé. Dans l'Epic 4, si une story implique des données sensibles, le CS doit inclure une clause de review sécurité explicite.

3. **Le guardrail JSX `&&` étendu aux expressions booléennes composées** — Le piège n'est pas juste `{condition && <Comp />}` — c'est aussi `{!a && !b ? <Comp /> : null}`. La règle doit être : **précalculer toute expression booléenne composée en variable const avant le JSX**.

4. **La consolidation post-rétro continue de porter ses fruits** — Story 2.0 a purgé Stripe et documenté les patterns critiques. Epic 3 a bénéficié directement : R2 patterns, Auth.js v5, tier-config, subscription-access. L'investissement en consolidation épique est ROI positif.

5. **Les stories de visibilité/tier sont les plus sensibles** — Story 3.4 (visibilité tier + teasers publics) est la story la plus critique sur le plan sécurité de l'Epic 3. Une fuite de données (montant, description, documents, contact) dans le mode public aurait été un incident de sécurité.
