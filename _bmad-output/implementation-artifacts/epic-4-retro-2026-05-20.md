## Défis

1. **DealCard liens imbriqués (Story 4.2 CR BLOCKER)** — `TagChips` rendait des `<Link>` à l'intérieur du `<Link>` de carte. HTML invalide, hydration possible cassée. Le CR a FAIL, fix appliqué : tags hors du lien carte, non interactifs dans `DealCard`. Ce pattern revient : **card + Link wrapper + sous-composant interactif = nested anchors**. Le guardrail doit être systématique.

2. **Premium access gate manquant sur `/dashboard/matching` (Story 4.3 CR BLOCKER)** — La page matching ne vérifiait pas `getUserPremiumAccess`. Un membre non abonné pouvait voir les deals matchés via matching alors que `/dashboard/opportunities` les bloquait. Fuite de données via nouvelle route. Le CR a FAIL, patch appliqué. **Règle : toute nouvelle page dashboard DOIT répliquer les gates des pages sœurs.**

3. **Notifications dupliquées sur re-verify admin (Story 4.3 CR MAJOR)** — `isAllowedTransition()` permettait `current === next` (idempotent pass-through), et le trigger de notification ne gardait pas contre `currentStatus === "VERIFIED"`. Re-vérifier un deal déjà VERIFIED renvoyait les notifications matched. **Pattern : `if (newStatus === "VERIFIED" && currentStatus !== "VERIFIED")` pour les side effects.**

4. **dev.db commité (Story 4.4, récurent)** — Le DS subagent a de nouveau inclus `dev.db` dans son commit. Le `.gitignore` couvre `*.db` mais `git add -A` le contourne. Story 4.0 avait documenté ce guardrail mais il n'a pas été suivi par le DS. Le CR patch a revert le fichier.

5. **JSX `&&` pré-existants non nettoyés** — 5 occurrences `&&` restent dans `src/` hors du scope des stories 4.x. Chaque CR les signale comme « pré-existant, defer ». Ils s'accumulent et deviennent une dette technique.

6. **`POST /api/user/profile` efface les tags si champ absent** — Si un client API omet le champ `tags`, le `deleteMany` + `createMany` efface les tags existants. L'UI envoie toujours les tags, mais l'API n'est pas robuste pour les clients non-UI.

## Insights Clés

1. **Toute nouvelle page dashboard doit copier les gates des pages sœurs** — C'est un pattern systématique. `/dashboard/matching` a oublié `getUserPremiumAccess`. La prochaine nouvelle page dashboard (Epic 5, Epic 6) oubliera probablement aussi. Le CS doit inclure une checklist explicite : « Quelles gates d'accès les pages sœurs utilisent-elles ? Réplique-les. »

2. **Nested anchors est le piège récurrent des card components** — Depuis Story 4.2, c'est la 2e fois (la 1re était en Story 3.x implicitement). Le pattern « Card dans `<Link>` + sous-composant avec `<Link>` » est un anti-pattern HTML. La solution : passer `interactive={false}` aux sous-composants dans le contexte card, ou les déplacer hors du wrapper `<Link>`.

3. **L'idempotence des side effects doit être un réflexe** — Les notifications, emails, webhooks sur transitions de statut doivent TOUJOURS vérifier `currentStatus !== targetStatus`. Sans ce guard, tout endpoint admin "idempotent" peut spammer des side effects. Ce pattern doit être documenté dans les guardrails du projet.

4. **La consolidation post-rétrospective fonctionne** — Story 4.0 a éliminé 2 items de dette ouverts depuis Epic 1 et documenté 3 guardrails. C'est le ROI direct de la discipline : faire une story de consolidation avant chaque epic. Maintenir cette pratique.

5. **Le modèle `Notification` est maintenant un fondement** — Créé dans 4.3 (OPPORTUNITY_MATCHED), réutilisé dans 4.4 (OPPORTUNITY_INTEREST). Le modèle est minimal mais extensible. Epic 5 (reviews) et Epic 6 (admin emails) peuvent l'enrichir avec de nouveaux types sans refonte.

## Découvertes Significatives

### 1. Le pattern « nouvelle page dashboard = fuite de données potentielle »

Chaque nouvelle route sous `(dashboard)/` qui affiche des données membres doit implémenter les mêmes gates que les pages existantes. Le simple fait d'ajouter une route sans `getUserPremiumAccess` crée un canal de contournement. L'Epic 6 (dashboard admin) aura le même risque si les gates admin sont oubliés.

**Impact :** Le CS story template doit inclure une checklist de gates explicitement. Le CR doit vérifier la parité de gates entre nouvelles et anciennes pages.

### 2. Les nested anchors HTML sont un anti-pattern invisible

Les composants fonctionnent individuellement (DealCard wrappe dans `<Link>`, TagChips rend des `<Link>`). C'est leur composition qui crée le problème. Ni le DS ni un test visuel ne le détectent facilement — seul le CR avec contexte architectural le signale.

**Impact :** Le guardrail doit être architectural (documenté), pas seulement procédural. Une règle ESLint custom `no-nested-anchor` serait l'idéal, mais au minimum la documentation dans `architecture.md` est nécessaire.

### 3. Le matching déterministe par tags est suffisant pour le MVP

L'algorithme `countCommonTags` sans ML ni scoring opaque a livré un feed matching fonctionnel. Les membres voient les deals partageant au moins un tag, triés par pertinence. Pour un volume de 500 membres actifs, ce calcul en mémoire est instantané.

**Impact :** L'optimisation SQL/agrégation n'est nécessaire que si le volume dépasse plusieurs milliers de membres avec tags. Jusque-là, `matching.ts` est un helper pur facilement testable et maintenable.

### 4. Le modèle `OpportunityInterest` est un pattern d'engagement faible réutilisable

La combinaison `@@unique([userId, opportunityId])` + gestion P2002 + `Promise.allSettled` pour les side effects email est un pattern idempotent complet. Il peut servir pour d'autres signaux faibles : watchlist, bookmark, follow — sans réinventer la logique de déduplication.

**Impact :** Si Epic 7 ou au-delà ajoute d'autres signaux d'engagement, le pattern `OpportunityInterest` est le modèle de référence.
