## Défis

1. **Premium access gate manquant sur `/members/[id]` (Story 5.2 CR MAJOR)** — La page profil membre chargeait les données réputation et persistait Platinum awards sans vérifier `getUserPremiumAccess()` au préalable. Un membre non-abonné pouvait voir les reviews et déclencher la persistence Platinum. Le CR a patché directement en ajoutant le gate + tests de coverage. **Pattern récurrent : toute modification d'une page dashboard existante DOIT auditer les gates des pages sœurs.**

2. **JSX `&&` résiduels — nettoyage à répétition** — Malgré le guardrail architecture documenté depuis Epic 3 et les consolidations 4.0 et 5.0, chaque DS subagent réintroduit des `&&` en JSX. Story 5.0 a nettoyé 8 fichiers. Ce n'est pas un problème de documentation insuffisante — c'est un problème de discipline DS subagent. Le guardrail ne fonctionne que par vérification CR ou grep manuel.

3. **dev.db staging — toujours une tentation** — Story 4.4 avait commité `dev.db`. Story 5.0 a réaffirmé le guardrail git staging explicite. Le risque persiste tant que les DS subagents utilisent `git add -A`. `.gitignore` couvre `*.db` mais le flag `-A` le contourne.

4. **Migration Prisma `avatarUrl` — toujours différé** — L'item de dette technique `avatarUrl` data loss dans la migration `20260514112346_init` a été traité en Story 5.0 (audit + correction du mapping `@map`), mais le risque de data loss en production reste si la migration a déjà été appliquée. Ce n'est pas un bug résolu, c'est un risque documenté et différé à Epic 6.

5. **Scope creep potentiel dans les stories de display** — Story 5.3 (affichage public reviews) risquait de réimplémenter le backend de reviews. Le delta-scoping explicite dans le Dev Notes a bien contenu le scope, mais seulement parce que le CS story l'a documenté. Sans cette mention, le DS aurait pu créer des endpoints dupliqués.

## Insights Clés

1. **Une modification de page dashboard existante = audit de gates obligatoire** — Ce n'est pas seulement les *nouvelles* pages qui manquent des gates (Epic 4, `/dashboard/matching`). Les *modifications* de pages existantes peuvent aussi introduire des données sensibles sans gate. Story 5.2 a ajouté de la réputation persistante à `/members/[id]` sans premium gate. Le CR doit vérifier les gates sur TOUTE page modifiée, pas seulement les nouvelles pages.

2. **Le pattern consolidation post-rétrospective est rentable** — Story 5.0 (consolidation) a payé les 3 items de dette technique de la rétro Epic 4 et documenté 2 guardrails. Résultat : Stories 5.1, 5.2, 5.3 ont fonctionné sur un codebase plus propre. Le ROI est mesurable : 0 blocker de type JSX/tags dans Epic 5 (vs 3 blockers en Epic 4).

3. **Les guardrails dans architecture.md ne contrôlent pas les DS subagents** — Le guardrail JSX `&&` est documenté depuis Epic 3. Chaque DS subagent l'ignore. La seule protection efficace est : (a) l'instruction explicite dans le DS dispatch context, et (b) le CR grep check. Les Dev Notes / architecture.md sont utiles pour les humains, pas pour les LLM agents avec fresh context.

4. **Platinum persistence idempotent est un pattern réutilisable** — `ensurePlatinumAwarded` vérifie `platinumAwardedAt === null` avant d'écrire, et ne supprime jamais le badge si le score baisse. Ce pattern "award once, never revoke" est applicable à d'autres badges ou certifications futures : une fois obtenu, le badge reste visible avec un indicateur "à maintenir".

5. **Le modèle `Review` avec `@@unique([reviewerId, opportunityId])` est extensible** — La contrainte unique permet les reviews mutuelles (reviewer A → reviewee B et reviewer B → reviewee A sur la même opportunity) tout en empêchant les doublons par reviewer. Ce pattern couvre les besoins actuels et est extensible pour les reviews sur deals clôturés ou les reviews entre membres connectés.

## Découvertes Significatives

### 1. Les modifications de pages existantes sont un canal de fuite de données aussi dangereux que les nouvelles pages

Story 5.2 a ajouté de la réputation persistante à `/members/[id]` sans premium gate — la page existait déjà et avait un gate, mais les données ajoutées (reviews, Platinum persistence) étaient accessibles sans gate. Le pattern de risque n'est pas seulement « nouvelle page sans gate » mais « page existante avec nouvelles données sans gate audité ».

**Impact :** Le CR doit vérifier les gates sur TOUTE page dashboard modifiée dans le diff, pas seulement les nouvelles pages. Le CS doit inclure un task explicite : « Vérifier que les gates d'accès couvrent les nouvelles données ajoutées. »

### 2. Le delta-scoping dans les stories de display est un pattern efficace

Story 5.3 (affichage public des reviews) a été scoppée comme un delta : « Ne pas recréer le backend de reviews, étendre le display existant. » Sans cette instruction explicite dans le CS, le DS aurait pu créer des endpoints dupliqués ou des modèles redondants. Le delta-scope doit être documenté dans chaque story de display qui suit une story de persistance.

**Impact :** Pour les futures stories de display/enrichissement, le CS doit toujours auditer le codebase existant et documenter ce qui est déjà fait vs ce qui reste à implémenter.

### 3. `calculateReliabilityScore` avec poids extensible est un bon design

L'implémentation utilise un helper avec poids par défaut `1` par review, ce qui permet d'ajouter des poids par tier/reviewer/couverture sans changer le code UI. Le helper est centralisé dans `src/lib/reputation.ts`, testé isolément, et consommé par les pages profil et deal sans duplication.

**Impact :** Si le produit demande des poids variables (reviews de Platinum pèsent plus, reviews récentes pèsent plus), le helper supporte cet ajout sans refonte des pages consommatrices.

### 4. La rétro-action de consolidation réduit les blockers drastiquement

Epic 4 avait 3 blockers en CR (nested links, missing premium gate, duplicate notifications). Epic 5 avait 1 blocker (premium gate sur page modifiée). La différence : Story 5.0 a consolidé 3 items de dette technique avant que les stories produit ne commencent. Les JSX `&&`, le tags absent, et les guardrails architecture étaient tous résolus avant 5.1.

**Impact :** La consolidation post-rétrospective n'est pas un luxe — c'est un investissement qui paie en blockers évités dans l'epic suivant.
